# 统计系统增强：能力雷达图 + 搭档分析

## 概述

在现有统计页（角色胜率 + 勋章）基础上，新增两大功能模块：
1. **能力雷达图**：好人/坏人各 6 维能力画像
2. **搭档分析**：称号卡片 + 配对矩阵

## 页面结构

统计页重构为 **首页 + Tab** 布局：

### 首页（滚动浏览，无需切 tab）
1. **个人概览卡片**：总场次、总胜率、连胜/连败
2. **能力雷达**：好人/坏人两张雷达图，可切换
3. **搭档称号**：6 张称号卡片
4. **搭档矩阵**：所有搭档列表，可展开详情

### Tab 区
- **角色胜率**：现有 8 角色详细数据（迁移自当前页面）
- **勋章墙**：现有勋章统计（迁移自当前页面）

全局 PvP/PvE 切换开关，所有 tab 共享。

## 一、能力雷达图

好人和坏人各一张雷达图，6 个维度。前端用 Canvas 绘制。

### 正义六维

| 维度 | 算法 | 数据来源 |
|------|------|----------|
| 识人能力 | 非梅林时，含坏人队伍投反对的比例 | voteHistory + assignments |
| 领袖力 | 当队长时发车成功率（队伍被投票通过） | voteHistory.leaderId + approved |
| 表水能力 | 非队长时被选入队伍的频率 | voteHistory.team（排除自己当队长的轮次） |
| 挡刀能力 | 非梅林时被刺客选为刺杀目标的比例 | assassination.targetId + assignments |
| 躲刀能力 | 梅林牌时未被刺杀的比例 | assassination.hit（role=梅林的局） |
| 好人胜率 | 正义阵营时的胜率 | game_participants.result（faction=good） |

### 邪恶六维

| 维度 | 算法 | 数据来源 |
|------|------|----------|
| 冲锋能力 | 送队友（坏人）上车的成功次数 | voteHistory.team 中含其他坏人 + approved |
| 隐秘性 | 在队伍中藏票（该出失败票时未出）的比例 | missionHistory.missionVotes（evil 出 false 的比例） |
| 表水能力 | 非队长时自己能上车的频率 | voteHistory.team（排除自己当队长的轮次） |
| 刺杀能力 | 带刀（刺客）时刺中梅林的比例 | assassination.hit（role=刺客的局） |
| 破坏力 | 出失败票且该任务确实失败的比例 | missionHistory.missionVotes + success |
| 坏人胜率 | 邪恶阵营时的胜率 | game_participants.result（faction=evil） |

### 算法细节

**分数归一化**：每个维度算出原始比率（0~1），再映射到 0~100 分。

**最低样本量**：某维度涉及的局数 < 3 时，该维度显示为灰色/虚线，不参与评分。

**角色难度加权**：胜率维度按角色基准胜率做偏移修正（如奥伯伦基准胜率低，赢了加分更多）。暂不实现，v1 用原始胜率。

**特殊处理**：
- 挡刀能力：只统计有刺杀阶段的局，且自己不是梅林
- 躲刀能力：只统计自己是梅林且有刺杀阶段的局
- 刺杀能力：只统计自己是刺客的局
- 隐秘性：evil 在队伍里出成功票（不暴露自己）的比例

## 二、搭档分析

### 第一层：称号卡片

6 张卡片，横向可滑动浏览：

| 称号 | 计算规则 | 最低门槛 |
|------|----------|----------|
| 黄金搭档 | 同阵营时胜率最高的搭档 | ≥ 3 局同阵营 |
| 最佳狼队友 | 同为坏人时胜率最高 | ≥ 3 局同为坏人 |
| 最佳骑士搭档 | 同为好人时胜率最高 | ≥ 3 局同为好人 |
| 最佳梅林&派西 | 你梅林他派西（或反过来）时胜率最高 | ≥ 2 局该组合 |
| 天生冤家 | 对面阵营时你胜率最低（最怕的对手） | ≥ 3 局对面阵营 |
| 最坑队友 | 同阵营时胜率最低 | ≥ 3 局同阵营 |

每张卡片显示：称号名、搭档昵称、搭档头像、胜率、局数。

### 第二层：搭档矩阵

按总同局次数降序的列表，每人一行：

**收起状态**：头像、昵称、总同局数、同阵营胜率

**展开详情**：
- 同为好人时：局数 / 胜率
- 同为坏人时：局数 / 胜率
- 对面阵营时：局数 / 你的胜率
- 最常角色组合（前 3）

## 三、数据层设计

### 服务端 API

扩展 `FETCH_ROLE_STATS` 响应，新增两个字段：

```javascript
{
  pvp: {
    // ... 现有字段 ...
    radar: {
      good: { recognition: 85, leadership: 72, trustworthiness: 60, shield: 45, dodge: 80, winRate: 68 },
      evil: { charge: 70, stealth: 55, trustworthiness: 65, assassination: 50, destruction: 78, winRate: 62 }
    },
    partners: {
      titles: [
        { type: 'golden', phone: '138xxx', nickname: '小明', avatar: '🐺', winRate: 78, games: 12 },
        // ... 其他称号
      ],
      matrix: [
        {
          phone: '138xxx', nickname: '小明', avatar: '🐺',
          totalGames: 20,
          sameTeam: { games: 14, wins: 10, winRate: 71.4 },
          sameGood: { games: 8, wins: 6, winRate: 75 },
          sameEvil: { games: 6, wins: 4, winRate: 66.7 },
          opponent: { games: 6, wins: 2, winRate: 33.3 },
          topCombos: [
            { myRole: '梅林', theirRole: '派西维尔', games: 3, wins: 3 },
            // ...
          ]
        },
        // ...
      ]
    }
  },
  pve: { /* 同结构 */ }
}
```

### 计算逻辑

在 `server/history.js` 中新增两个函数：

**`_buildRadar(phone, excludeAI)`**：
1. 查询该玩家所有已完成对局的 game_id
2. 逐局读取 payload JSON
3. 按好人/坏人分别累加各维度的分子分母
4. 算出 6 维分数（0~100）

**`_buildPartners(phone, excludeAI)`**：
1. 查询该玩家参与的所有对局
2. 按同局的其他玩家 phone 分组
3. 对每个搭档累计：同阵营/对面阵营的局数和胜负
4. 按规则选出 6 个称号
5. 返回称号 + 矩阵

### 性能考虑

**payload 解析开销**：每次请求需遍历所有历史对局的 payload JSON。

**优化策略（v1）**：
- 限制最多分析最近 100 局（覆盖绝大多数玩家）
- payload 只取需要的字段（voteHistory, missionHistory, assassination, players）
- 称号和矩阵在一次遍历中同时计算

**后续优化（v2，如有性能问题）**：
- 新增 `player_stats_cache` 表，每局结束时增量更新
- 缓存 TTL 机制

## 四、前端渲染

### 雷达图（Canvas）

- 420rpx × 420rpx 正六边形
- 好人蓝色系，坏人红色系
- 顶部切换按钮：正义 / 邪恶
- 每个顶点标注维度名 + 分数
- 样本不足的维度用虚线 + 灰色

### 称号卡片

- 横向滚动，每张卡片宽 280rpx
- 卡片内容：称号图标 + 搭档头像 + 昵称 + 胜率 + 局数
- 数据不足时显示"暂无数据"

### 搭档矩阵

- 纵向列表，默认收起
- 点击展开详情面板
- 按同局次数降序
- 列表项：头像 + 昵称 + 同局数 + 同阵营胜率进度条
